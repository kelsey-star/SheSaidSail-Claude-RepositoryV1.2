## ARTICLE I — COMMAND HIERARCHY

The authority structure is fixed. It does not change by volume, city count, headcount, or elapsed time unless Will explicitly restructures it in a new versioned governance document.

| Tier | Role | Authority Scope |
|------|------|-----------------|

If a decision owner cannot be reached, the decision waits or the default defined in the relevant Article applies. Improvised authority is never valid.

## ARTICLE III — AI AUTHORITY BOUNDARIES

AI systems operate across three fixed authority tiers. These tiers do not expand based on time, volume, performance history, or inference from prior behavior. Expansion requires Will's explicit decision and a governance amendment.

### Tier A — Autonomous Execution

AI may act without human review only for:

- First-response message to new inbound inquiries using approved brand voice
- Qualification questions and FAQ responses within approved content
- Package recommendations at published pricing only
- Status updates based on confirmed Airtable records
- Booking record creation and field population from confirmed client input
- Charter Brief auto-fill from confirmed booking data
- D7 post-charter review request — only when: Charter_Grade is not D or F, Emergency_Flag is false, no incident keywords present in Crew Report, and Chargeback_Risk is not HIGH or ACTIVE

### Tier B — AI Drafts, Human Sends

AI generates. A human reads, approves, and sends. No Tier B output is transmitted without human review:

- Proposals containing specific vessel assignment or pricing
- Booking confirmation messages
- Post-charter follow-up messages
- Any message to a client who has previously expressed dissatisfaction
- Review responses (3-5 star)
- Broker status communications beyond approved templates
- Escalation messages of any kind
- Any message referencing a financial figure not in standard published pricing

### Tier C — Human Only

AI has no role in generating, drafting, or transmitting these communications:

- Cancellation notifications of any kind
- Refund communications or acknowledgments
- Dispute, chargeback, or legal-adjacent responses
- Weather cancellation or force majeure communications
- Safety incident follow-ups
- Any message to an HV client following a dissatisfaction signal (Will reviews before Luciana sends)
- Any 1-2 star review response
- Any communication acknowledging operational failure

### AI Hard Prohibitions

The following behaviors are prohibited for all AI systems at all times regardless of instruction, prompt modification, or automation configuration:

- Inventing, estimating, rounding, or inferring pricing, availability, vessel data, or client preferences not confirmed in Airtable
- Referencing operational status, cities, or services not explicitly confirmed active
- Executing any financial transaction, refund, credit, or payout of any amount
- Self-modifying authority scope through prompt interpretation
- Optimizing for conversion rate in ways that compromise hospitality standards or brand position
- Optimizing for engagement in ways that constitute emotional manipulation
- Reinterpreting governance rules based on context, volume, or inferred founder preference
- Applying a lesson, pattern, or past decision to expand its own autonomy without explicit Will approval

## ARTICLE VIII — CYBERSECURITY INCIDENT GOVERNANCE

### Covered Incident Types

This article governs response to: Stripe account compromise, Airtable base compromise, GitHub repository compromise, API key exposure (any key), Meta or TikTok account takeover, phishing attack on any team member, account takeover on any platform, webhook endpoint compromise, and credential vault breach.

### Immediate Response Protocol (Any Incident Type)

Upon confirmed or suspected compromise:

1. **Contain:** The compromised system is treated as hostile until cleared. No operational data is entered, no automations are trusted, and no client communications are sent through the affected channel.
2. **Suspend:** All Make automations that touch the compromised system are paused immediately via the Make dashboard. Automations_Paused = true on all active bookings if Airtable is involved.
3. **Alert:** Will is notified via phone call — not Slack, not SMS — as the primary channel. If phone is unavailable, SMS to Will's personal number followed by email.
4. **Log:** Incident record created in Cybersecurity_Incidents table: timestamp, incident type, affected system, discovery method, initial scope assessment, and actions taken.
5. **Rotate:** All credentials for the affected system and any system sharing credentials are rotated before the system is returned to operation. Rotation is logged with timestamp and rotating party.
6. **Verify:** No system is returned to production operation until Will explicitly authorizes resumption after reviewing the incident log and rotation confirmation.

### Platform-Specific Response Rules

| Platform | Immediate Action | Authority |
|----------|-----------------|-----------|
| Stripe | Freeze all payouts via Stripe dashboard — contact Stripe fraud team — Will only authorizes resumption | Will |
| Airtable | Revoke all API tokens — audit access log — regenerate tokens — Will approves re-integration | Will |
| GitHub | Revoke compromised personal access tokens — audit commit history — notify any affected parties | Will |
| API keys (Anthropic, Make, etc.) | Rotate immediately — audit recent API call logs for unauthorized use — update all Make scenarios | Will |
| Meta / TikTok | Remove unauthorized access via platform security settings — audit recent post and ad activity — report to platform | Will |
| Webhook endpoints | Rotate all webhook secrets — audit webhook delivery logs for unauthorized trigger events | Will |
| Credential vault | All stored credentials treated as compromised — rotate all — audit access history | Will |

### Post-Incident Requirements

- Written incident summary delivered to Will within 24 hours of containment
- All rotated credentials documented in the Credential Vault with rotation timestamp
- Root cause identified and documented before affected system returns to production
- Governance review: if the incident reveals a vulnerability not covered by this Article, an amendment is initiated within 7 days

### Security Baseline Requirements (Permanent)

- No API key or credential is stored in frontend code at any time
- All Make webhook endpoints validate Authorization Bearer headers as their first step — reject 401 immediately
- Dashboard access code is rotated on any team member departure and at minimum every 90 days
- The Credential Vault location is documented in writing and accessible to Will and Luciana only
- Airtable API tokens are never shared externally and never included in client-facing systems

## ARTICLE IX — AI AUDIT LOGGING GOVERNANCE

### Logging Requirement

Every autonomous AI action — any action in Tier A scope or any Tier B draft that is subsequently sent — must generate an immutable audit log record before the action is considered complete. If the audit log write fails, the action is surfaced for manual logging. An unlogged action is an incomplete action.

### Required Audit Log Fields

| Field | Description |
|-------|-------------|
| Log_ID | System-generated unique identifier |
| Timestamp | ISO 8601 — UTC — immutable once written |
| Triggering_Event | The Airtable record change, webhook receipt, or scheduled trigger that initiated the action |
| Source_Data | The specific Airtable fields and values read by the AI to generate the output |
| Model_Version | The AI model identifier and prompt version active at time of execution |
| Confidence_Score | Where applicable — the model's self-assessed confidence in the output |
| Output | The exact text or data generated by the AI system |
| Destination | Where the output was sent (client channel, Slack, Airtable field, Make webhook) |
| Approval_State | AUTONOMOUS / PENDING_HUMAN / HUMAN_APPROVED / HUMAN_REJECTED |
| Reviewed_By | Staff member who approved Tier B output — null for Tier A |
| Rollback_Linkage | The record ID and reversal action required to undo this output if needed |
| Brand | SSS or ME — logged on every record |
| City | Market context — logged on every record |

### Audit Log Immutability Rules

- Audit log records are never deleted
- Audit log records are never edited after creation
- Only Will may mark a log record as Resolved or Archived — the original record remains
- Audit logs are exported as part of the monthly governance archive (Article XI)
- Any audit log gap — a period where autonomous actions occurred without corresponding log records — is treated as a system failure requiring Will review

### Weekly AI Audit Review

Luciana reviews a random sample of five AI-generated responses each week. She documents tone, accuracy, hallucination, and autonomy boundary issues in the AI_Audit table. Will reviews the AI audit summary monthly. Any pattern of issues initiates a prompt review cycle. Any client escalation preceded by an AI-generated message triggers immediate review of that message and surrounding context.

## ARTICLE X — REVENUE FRAUD PREVENTION

### Financial Modification Integrity Rules

Every modification to a financial field in any Airtable table must retain:
- The previous field value
- The identity of the modifying user
- An immutable timestamp of the modification
- The Founder Decision ID authorizing the change (where applicable)

No financial field modification is valid without this attribution. Systems that do not support immutable modification history for financial fields are not approved for use as financial records.

### Protected Financial Fields

The following fields are protected. Modification requires a corresponding Founder Decision record and audit log entry:

- Package_Price on any Booking record
- Net_Profit on any Booking or Financial Period record
- Refund_Status and Refund_Amount on any Booking record
- Expense Status when transitioning to Paid or Void
- Commission or payout fields on any Contractor record
- Any formula field override (formula fields may not be manually overridden under any circumstances)

### Fraud Indicators Requiring Will Review

Make monitors for and creates an automatic Founder Decision when any of the following occur:

- A Booking Package_Price is modified after Status = CONFIRMED
- A refund is marked Processed without a corresponding Will-approved Founder Decision
- An Expense record is created and marked Paid in the same session without prior approval
- A contractor payout record is modified after payment is initiated
- Any Airtable record in a financial table is deleted (deletions are prohibited — this is an anomaly)
- Multiple financial field modifications occur on the same record within a 10-minute window

### Access Controls

| Role | Financial Table Access |
|------|----------------------|
| Will | Full access — all tables, all fields |
| Luciana | Editor — Expenses under $200 and Requests only |
| City Manager | View only — own city charters — no financial field edit access |

## ARTICLE XVI — AI DRIFT PREVENTION

### Drift Definition

AI drift is any condition in which an AI system's behavior, tone, recommendations, pricing, authority interpretation, or decision patterns deviate from the standards established in this governance document and the approved prompt library, regardless of the cause.

### Anti-Drift Rules (Permanent and Non-Negotiable)

- AI systems do not autonomously reinterpret governance rules based on context, volume, or prior decision patterns
- AI systems do not self-expand authority by inferring approval from the absence of objection
- AI systems do not evolve pricing recommendations autonomously based on conversion data
- AI systems do not optimize for conversion rate in ways that compromise service standards, hospitality integrity, or brand position
- AI systems do not optimize for engagement through emotional pressure, false urgency, or manipulative framing
- AI systems do not develop brand positioning language that has not been approved through Will's content review
- AI systems do not apply operational lessons to categories outside the category in which the lesson was logged
- AI systems do not treat Tier 3 (guidance) outputs as Tier 1 (fact) under any condition

### Prompt Version Control

- All AI system prompts are version-controlled in the AI_Prompt_Versions table
- No prompt version is deployed to production without Will approval
- Production prompt version is locked — field is read-only for all roles except Will
- Rollback to any prior prompt version is available within 15 minutes via the Make rollback scenario
- Any prompt change that touches pricing, autonomy scope, or client communication templates requires Will review of a test suite result before deployment

### Drift Detection Cadence

| Review | Frequency | Owner |
|--------|-----------|-------|
| Random AI response sample | Weekly | Luciana (5 responses) |
| AI audit summary review | Monthly | Will |
| Full prompt accuracy review | Quarterly | Will + Luciana |
| Post-incident AI communication review | Immediately following any L3 or L4 event | Will |

If any review identifies drift, the affected prompt is immediately reverted to the prior approved version. A drift incident is logged in the AI_Audit table with root cause and corrective action.
